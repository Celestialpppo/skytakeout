# 苍穹外卖订单幂等功能分析与简历话术

## 1. 结论

基于当前工作区代码来看，这个功能**已经实现了**，不是只有方案文档。

而且实现的不是单点“加个 Redis 锁”，而是完整的三层防线：

1. **请求级幂等**
   下单用 `submitToken` + Redis 原子删除防重复提交。
2. **并发级串行化**
   下单、支付、支付回调都用了 `Redisson RLock`。
3. **状态级与存储级兜底**
   支付回调走订单状态机，最终落库时再用 `update ... where status = oldStatus` 做 CAS 条件更新。

所以如果别人问你“你这个 Redis 分布式锁 + 状态机的订单幂等做了吗”，答案是：

**做了，而且下单幂等、支付发起幂等、支付回调幂等三段都已经落地。**

## 2. 代码证据

### 2.1 下单幂等已实现

证据：

- `sky-server/src/main/java/com/sky/controller/user/IdempotentController.java`
  提供 `/user/idempotent/submitToken` 接口生成一次性提交令牌。
- `sky-server/src/main/java/com/sky/service/impl/IdempotentServiceImpl.java`
  用 Redis 存 `order:submit:token:{userId}:{token}`，并在提交时原子删除。
- `sky-server/src/main/java/com/sky/service/impl/OrderServiceImpl.java`
  `submit()` 中执行了完整链路：
  - 先查 `order_request_log`
  - 再校验并删除 Redis token
  - 再按 `userId + cartDigest` 加 Redisson 锁
  - 再写 `order_request_log`
  - 成功后回填 `orderId/orderNumber`
- `sky-server/src/main/java/com/sky/mapper/OrderRequestLogMapper.java`
  支撑下单幂等日志查询和状态更新。
- `sky.sql`
  已新增 `order_request_log` 表，并加了唯一索引 `uk_order_request_user_token (user_id, submit_token)`。

### 2.2 支付发起幂等已实现

证据：

- `sky-server/src/main/java/com/sky/service/impl/OrderServiceImpl.java`
  `payment()` 中按 `order:pay:lock:{orderNumber}` 加锁。
- 同一个订单支付时：
  - 若 `payment_txn` 已成功，直接判定已支付
  - 若是待回调状态，直接幂等返回
  - 若失败，则刷新 `payRequestId` 后允许重试
- `sky-server/src/main/java/com/sky/mapper/PaymentTxnMapper.java`
  支撑按订单号查询支付流水、更新支付结果。
- `sky.sql`
  已新增 `payment_txn` 表，并加唯一索引 `uk_payment_txn_order_number`。

### 2.3 支付回调幂等已实现

证据：

- `sky-server/src/main/java/com/sky/controller/user/PayCallbackController.java`
  已提供微信、支付宝回调入口。
- `sky-server/src/main/java/com/sky/service/impl/PayCallbackServiceImpl.java`
  回调处理链路是完整的：
  - 先提取 `orderNumber + transactionId`
  - 生成 `callbackId`
  - 查询 `payment_callback_log`
  - 如果已成功处理，直接幂等返回
  - 对同一订单加支付锁
  - 写/更 `payment_callback_log` 为 `PROCESSING`
  - 查询订单和支付流水
  - 只有订单还是待支付时，才触发状态机 `PAY`
  - 成功后把 `payment_txn` 和 `payment_callback_log` 更新为成功
- `sky-server/src/main/java/com/sky/mapper/PaymentCallbackLogMapper.java`
  支撑回调幂等日志。
- `sky.sql`
  已新增 `payment_callback_log` 表，并加唯一索引 `uk_payment_callback_id`。

### 2.4 订单状态机兜底已实现

证据：

- `sky-server/src/main/java/com/sky/config/OrderStateMachineConfig.java`
  已定义 `PENDING_PAYMENT -> TO_BE_CONFIRMED` 的 `PAY` 事件流转。
- `sky-server/src/main/java/com/sky/service/state/OrderStateContext.java`
  已把状态路由做成无状态上下文。
- `sky-server/src/main/java/com/sky/service/state/PendingPaymentState.java`
  支付成功不是直接随便改状态，而是：
  - 状态机接受 `PAY`
  - 再调用 `orderMapper.updateWithCondition(order, oldStatus)` 做条件更新
- `sky-server/src/main/resources/mapper/OrderMapper.xml`
  `updateWithCondition` 是 `where id = #{id} and status = #{oldStatus}`。

这说明你现在的设计不是“锁即幂等”，而是：

**Redis 锁拦并发，状态机控合法流转，数据库 CAS 做最后兜底。**

## 3. 借鉴 CongoMall 了吗

## 结论

**借鉴了思路，但不是直接照搬代码。**

更准确地说：

## 4. 借鉴了什么地方

### 4.1 借鉴了“幂等是基础能力，不是零散 if 判断”

`congomall/README.md` 明确写了它有：

- 分布式锁
- 分布式幂等组件
- 分布式 ID
- 消息队列

你这版 `sky-take-out` 也明显采用了类似思路：

- 单独抽了 `IdempotentService`
- 单独建了 `order_request_log`
- 单独建了 `payment_txn`
- 单独建了 `payment_callback_log`

这说明你不是把幂等写死在 Controller，而是把它抽成了一个可复用的能力层。

### 4.2 借鉴了“按业务键加 Redisson 锁”的思路

CongoMall 里有明显的业务键加锁方式，例如：

- `order:canal:order_sn_xxx`
- `order:status-reversal:order_sn_xxx`
- 购物车也按用户+商品维度生成锁 key

你这边对应借鉴为：

- `order:submit:lock:{userId}:{cartDigest}`
- `order:pay:lock:{orderNumber}`

这属于明显的“按业务对象粒度锁定”的工程思路借鉴。

### 4.3 借鉴了“不要只靠前端防重，要靠后端幂等兜底”

CongoMall 下单接口直接用了 `@Idempotent` 注解。
你虽然没有用它的注解实现，但本质目标一致：

- 防重复请求
- 防并发重入
- 防状态重复推进

只是你的实现方式换成了：

- Redis token
- 幂等日志表
- Redisson 锁
- 状态机 + 条件更新

## 5. 没借鉴什么地方

这些地方基本可以判断**没有直接借鉴 CongoMall 的实现代码**：

1. **没有用 CongoMall 的 `@Idempotent` 注解**
   你这边是自己写的 `IdempotentService` + 日志表方案。
2. **没有复用 CongoMall 的 Starter**
   代码里没有接入 `congomall-idempotent-spring-boot-starter`。
3. **没有照搬它的 DDD 聚合/仓储结构**
   你这里还是 `Controller + ServiceImpl + Mapper` 的单体结构。
4. **没有把支付做成独立微服务**
   CongoMall 是订单、支付分服务，你这里还是单体项目内实现。
5. **没有引入 MQ / Seata / 分布式事务**
   你这版重点是订单幂等和并发一致性，不是分布式事务编排。

所以你面试里最好不要说“参考 CongoMall 直接照着抄的”，更稳的说法是：

**参考了 CongoMall 在分布式锁、幂等抽象和业务键锁粒度上的设计思想，结合苍穹外卖单体架构，重新做了适合本项目的轻量化实现。**

## 7. 如果我是面试官，我会怎么问

### 7.1 你为什么不能只用 Redis 分布式锁做幂等？

**答：**
因为锁只能解决“同一时刻只有一个线程执行”，不能解决“这个请求之前是不是已经成功过”。如果锁过期、服务重启、请求重试，单靠锁仍然可能重复执行。所以必须叠加幂等标识和持久化记录。我的方案里下单是 `submitToken + order_request_log`，支付回调是 `callbackId + payment_callback_log`，锁只是第一道门。

### 7.2 为什么下单要同时用 submitToken 和 order_request_log？

**答：**
`submitToken` 是入口层快速防重，成本低、响应快；`order_request_log` 是持久化兜底，防 Redis key 过期、网络重试或极端并发。前者解决大部分重复提交，后者保证最终可回放、可追踪、可恢复，两者一起才稳定。

### 7.3 为什么支付成功不能以前端调支付接口为准？

**答：**
因为前端点击支付只是“发起支付”，不代表第三方已经扣款成功。真正可靠的成功信号只能是第三方支付平台回调。所以我在支付发起阶段只写 `payment_txn`，不推进订单状态；只有回调成功后才通过状态机把订单从待支付改成待接单。

### 7.4 你为什么还要加状态机，直接 update status 不行吗？

**答：**
直接 update 容易出现非法状态流转，比如已取消订单又被支付成功、已支付订单再次支付。状态机把允许的流转路径固定下来，比如 `PENDING_PAYMENT -> TO_BE_CONFIRMED` 才能触发 `PAY`，这样可以保证业务规则不被并发或脏请求破坏。

### 7.5 既然有状态机了，为什么还要 `update where status = oldStatus`？

**答：**
因为状态机主要是在内存里判断业务流转是否合法，真正落库时仍然可能被并发线程抢先更新。`update where status = oldStatus` 相当于数据库 CAS，是最后一道门。即使两个线程都走到更新阶段，也只能有一个更新成功。

### 7.6 支付回调重复通知时你怎么处理？

**答：**
我先用 `orderNumber + transactionId` 生成 `callbackId`，先查 `payment_callback_log`。如果这个回调已经成功处理过，就直接返回 success；如果没处理过，再对订单号加分布式锁串行执行。这样不管第三方回调几次，最终都只会有一次真正推进订单状态。

### 7.7 为什么支付流水表要单独建，而不是只改订单表？

**答：**
订单表只适合存最终业务状态，不适合记录支付发起、回调、失败原因、三方交易号这些支付过程信息。单独建 `payment_txn` 之后，支付发起幂等、失败重试、回调对账、问题排查都会更清楚，职责也更分离。

### 7.8 你的锁粒度为什么这么设计？

**答：**
下单我按 `userId + cartDigest` 加锁，是为了把粒度控制在“同一用户的同一份购物车快照”，避免把用户所有操作都串行化；支付我按 `orderNumber` 加锁，因为支付和回调的核心竞争对象就是同一笔订单。

### 7.9 如果 Redis 突然挂了，这套方案还安全吗？

**答：**
会退化，但不会完全失控。因为我还有数据库唯一索引、幂等日志表和条件更新做兜底。Redis 不可用时，并发拦截能力会下降，但同一 token 重复下单、同一 callback 重复处理、同一状态重复推进，最终仍能靠数据库层约束住。

### 7.10 这个方案最大的亮点是什么？

**答：**
亮点不是“用了 Redis 锁”，而是我把幂等拆成了三层：请求级防重、状态级约束、存储级兜底。这样讲出来更像工程方案，而不是一个零散技巧。
