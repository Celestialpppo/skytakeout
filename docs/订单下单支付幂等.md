# `sky-take-out` 订单下单/支付幂等与并发一致性改造方案

## 1. 目标

在 `sky-take-out` 现有订单模块基础上，补齐以下三层能力：

1. **下单幂等**：解决用户重复点击“提交订单”、前端超时重试、弱网重复请求导致的重复下单。
2. **支付幂等**：解决支付接口重放、第三方支付回调重复通知、人工补单重复触发导致的重复支付。
3. **数据库最后一道保护**：即使 Redis 锁失效、服务重启、并发穿透，也能保证订单状态只被合法更新一次。

本方案默认项目形态为：**单体 Spring Boot + MySQL + Redis**，并且订单状态已通过 **StateMachine** 建模。因此目标不是把项目过度改造成微服务，而是在现有结构上做一套“**够用、可讲、能落地**”的高质量增强方案。

## 2. 参考项目对比结论

### 2.1 CongoMall：适合作为“架构上限”参考

CongoMall 的公开说明直接写明：系统集成了 **分布式锁、分布式事务、消息队列**，并且提供了 **规约、缓存、幂等、分布式 ID** 等底层组件能力。对“订单 + 支付 + 幂等”的参考价值最大。  
**结论**：适合借鉴它的分层思路：
- 业务层：订单、支付、购物车
- 基础组件层：锁、幂等、ID、消息、事务
- 设计原则：把“幂等”做成通用能力，而不是散落在控制器里

### 2.2 mall / mall-swarm：适合作为“Redis + MQ + 订单流转”参考

`mall` 的 README 明确列出 **RabbitMQ** 和 **Redis** 作为核心后端技术；公开 issue 中还能看到其订单号生成逻辑使用 Redis 自增。`mall-swarm` 在此基础上进一步引入 **Seata** 做分布式事务管理。  
**结论**：适合借鉴它们的两点：
- **Redis** 用于高频状态/编号/缓存类能力；
- **MQ** 用于订单超时关闭、异步化解耦；
- **Seata** 只适合真正跨服务事务场景，不适合在 `sky-take-out` 单体项目里硬套。

### 2.3 litemall：适合作为“单体项目靠支付回调 + 状态字段维持流程”的参考

litemall 的 FAQ 明确说明：支付成功后需要微信回调 `payNotify` 才能把订单从“未付款”改成已支付；相关 issue 也能看到其核心是围绕“订单状态值”推进支付后逻辑。  
**结论**：适合借鉴它的“**支付必须以回调确认为准**”这一业务事实，但不适合作为分布式锁方案的主要参考，因为它更偏传统单体状态流转。

### 2.4 newbee-mall：适合作为“功能演进路径”参考

`newbee-mall` 基础版是单体商城；README 里说明 `newbee-mall-plus` 增加了 **秒杀、支付、Redis**，`newbee-mall-cloud` 增加了 **微服务 + Seata**。  
**结论**：适合借鉴它的演进路径：
- 基础版：单体事务 + 状态字段
- 增强版：Redis 应对高并发热点业务
- 云版：分布式事务只在系统拆分后再引入

### 2.5 综合判断

对于 `sky-take-out` 这种体量，最合理的方案不是一上来用 Seata，也不是只写一句“加 Redis 分布式锁”。

**推荐组合：**

- **下单幂等**：请求令牌 + Redis 分布式锁 + 唯一业务键
- **支付幂等**：支付流水表 + 回调幂等表 + Redis 分布式锁 + 状态机
- **最终一致性保护**：数据库条件更新（CAS）或乐观锁版本号
- **异步扩展能力**：可选接入 MQ 处理超时关单、支付后通知商家

这套方案最适合写进简历，也最贴合 `sky-take-out` 的实际架构。

## 3. 总体设计原则

### 原则一：不要把“锁”当成幂等的全部

Redis 锁只能限制“同一时刻只有一个线程处理”，但**不能天然替代幂等记录**。真正稳定的方案必须同时具备：

- 锁：限制瞬时并发
- 幂等键：识别“这是不是同一次业务请求”
- 状态机：判断“这个状态是否允许继续推进”
- 数据库 CAS：防止最终写入冲突

### 原则二：支付以“回调幂等”为核心，不以前端点击为核心

前端“去支付”只是发起支付；**真正的支付成功，应以第三方支付回调通知为准**。

### 原则三：数据库必须有兜底

Redis 是第一道门，状态机是第二道门，**数据库条件更新是最后一道门**。

## 4. 目标架构

```text
用户点击提交订单
    -> 校验 submitToken
    -> Redis 锁(order:submit:{userId}:{cartDigest})
    -> 生成幂等请求号 requestNo
    -> 创建订单（PENDING_PAYMENT）
    -> 记录 order_request_log
    -> 释放锁

用户发起支付
    -> Redis 锁(order:pay:{orderNo})
    -> 查询订单状态
    -> 若已支付，直接返回“已处理”
    -> 若待支付，调用支付网关
    -> 记录 payment_txn（待回调）
    -> 释放锁

支付平台回调
    -> 唯一键校验 callbackId / outTradeNo
    -> Redis 锁(order:pay:callback:{orderNo})
    -> 订单状态机校验 PENDING_PAYMENT -> TO_BE_CONFIRMED
    -> 数据库条件更新 where id=? and status=PENDING_PAYMENT
    -> 写入 payment_txn / callback_log
    -> 返回 success
```

## 5. 下单幂等方案

## 5.1 方案目标

解决以下问题：

- 用户连续点击“提交订单”两次
- 前端请求超时后自动重试
- Nginx/网关层重复转发
- 同一购物车短时间内重复生成多笔订单

## 5.2 技术组合

### 必选技术

1. **Redis 一次性提交令牌（submitToken）**
2. **Redis 分布式锁（建议 Redisson RLock）**
3. **订单请求日志表（order_request_log）**
4. **唯一业务键：`request_no`**

### 为什么推荐 Redisson，而不是手写 SET NX PX

可以手写 `SET key value NX PX 5000` + Lua 解锁，但对简历和工程落地来说，**Redisson 更稳、更省实现成本**：

- 自动续期
- API 语义清晰
- 避免误删别人的锁
- 面试时也更好解释

**结论**：
- 面试回答可以提到原理是 `SET NX PX + Lua`；
- 实际落地建议直接用 **Redisson**。

## 5.3 推荐流程

### 第一步：确认页生成 `submitToken`

进入订单确认页时，后端生成一个随机 `submitToken`，写入 Redis：

```text
key: order:submit:token:{userId}:{token}
TTL: 5min
```

前端提交订单时必须携带此 token。

### 第二步：Lua 脚本原子校验并删除 token

提交订单时先执行 Lua：

- token 存在：删除并返回成功
- token 不存在：视为重复提交

这样可以避免前端拿同一个 token 连续提交。

### 第三步：对“同用户 + 同购物车摘要”加锁

锁键建议：

```text
order:submit:lock:{userId}:{cartDigest}
```

其中 `cartDigest` 由以下字段摘要生成：

- 商品 id 列表
- 数量
- 口味/规格
- 收货地址 id
- 配送费/金额快照

这样比只按 `userId` 加锁更细，避免同一用户操作多个订单场景被串行化过度。

### 第四步：生成业务请求号 `requestNo`

提交订单时生成唯一请求号：

```text
requestNo = 雪花ID / UUID / Redis 自增序列
```

同时写入 `order_request_log`：

| 字段 | 含义 |
|---|---|
| request_no | 下单请求唯一号 |
| user_id | 用户ID |
| cart_digest | 购物车摘要 |
| order_id | 订单ID |
| status | INIT / SUCCESS / FAIL |
| create_time | 创建时间 |

并对 `request_no` 建唯一索引。

### 第五步：创建订单并清理购物车

订单创建事务中完成：

- 写订单主表
- 写订单明细表
- 清空购物车
- 更新 `order_request_log.status = SUCCESS`

## 5.4 关键点

- **submitToken** 解决“前端重复点击”
- **Redis 锁** 解决“瞬时并发进入服务端”
- **requestNo + request_log** 解决“请求重放识别”

三者缺一不可。

## 6. 支付幂等方案

## 6.1 方案目标

解决以下问题：

- 用户对同一订单多次点击支付
- 同一订单发起多次支付请求
- 第三方支付平台重复回调
- 商户补偿任务重复处理

## 6.2 技术组合

### 必选技术

1. **支付流水表 `payment_txn`**
2. **支付回调日志表 `payment_callback_log`**
3. **Redis 分布式锁**
4. **订单状态机**
5. **数据库条件更新 / 乐观锁**

## 6.3 数据表设计建议

### 1）支付流水表 `payment_txn`

| 字段 | 含义 |
|---|---|
| id | 主键 |
| order_id | 订单ID |
| order_no | 订单号 |
| request_no | 支付请求号 |
| out_trade_no | 商户支付单号 |
| channel_txn_no | 第三方交易流水号 |
| pay_channel | 支付渠道 |
| pay_status | INIT / PAYING / SUCCESS / FAIL |
| amount | 金额 |
| version | 乐观锁版本号 |
| create_time | 创建时间 |
| update_time | 更新时间 |

唯一索引建议：

- `uk_out_trade_no`
- `uk_channel_txn_no`（允许为空时需结合数据库特性设计）

### 2）支付回调日志表 `payment_callback_log`

| 字段 | 含义 |
|---|---|
| id | 主键 |
| order_no | 订单号 |
| out_trade_no | 商户支付单号 |
| channel_txn_no | 三方流水号 |
| callback_id | 回调唯一号/摘要 |
| raw_payload | 原始回调报文 |
| process_status | INIT / SUCCESS / IGNORE |
| create_time | 创建时间 |

唯一索引建议：

- `uk_callback_id` 或 `uk_channel_txn_no`

## 6.4 推荐流程

### A. 用户发起支付

1. 对 `order:pay:{orderNo}` 加 Redis 锁。
2. 查订单当前状态：
   - 若已支付/待接单/已完成：直接返回“已处理成功”；
   - 若是待付款：允许继续。
3. 生成 `outTradeNo`，写入 `payment_txn`。
4. 调用支付网关。
5. 返回支付参数给前端。

> 注意：这里不要在“发起支付请求”阶段就把订单状态改为已支付。

### B. 支付平台回调

1. 先对回调唯一键做幂等校验：
   - `callback_id` 不存在：进入处理；
   - 已存在：直接返回 success。
2. 再对 `order:pay:callback:{orderNo}` 加锁。
3. 查询订单状态：
   - 若已支付：直接返回 success（幂等成功）；
   - 若待付款：继续处理。
4. 调用状态机事件 `PAY`。
5. 执行数据库条件更新：

```sql
update orders
set status = 'TO_BE_CONFIRMED', pay_status = 1, pay_time = now()
where id = ? and status = 'PENDING_PAYMENT';
```

6. 更新 `payment_txn = SUCCESS`。
7. 记录 `payment_callback_log = SUCCESS`。
8. 返回第三方 success。

## 6.5 为什么支付幂等不能只靠状态机

状态机只能保证“**非法流转不通过**”，例如：

- 待付款 -> 已支付：允许
- 已支付 -> 再支付一次：不允许

但它不能单独解决：

- 支付回调重复投递
- 多实例并发处理同一回调
- 锁失效后数据库覆盖更新

所以必须叠加：

- 回调日志唯一键
- Redis 锁
- 数据库条件更新

## 7. 数据库最后一道保护

## 7.1 推荐做法：条件更新优先，版本号可选

对 `sky-take-out` 这种项目，**优先推荐条件更新（CAS）**，简单直接：

```sql
update orders
set status = ?, update_time = now()
where id = ? and status = ?;
```

### 优点

- 改造量小
- 不一定要全表引入 `version`
- 面试解释清晰
- 和状态机天然匹配

## 7.2 备选做法：乐观锁版本号

如果你想包装得更工程化，可以在订单表和支付流水表中加：

```sql
version int not null default 0
```

更新时：

```sql
update orders
set status = ?, version = version + 1
where id = ? and version = ?;
```

### 适用场景

- 同一行会被多个后台流程频繁修改
- 需要更细粒度的并发失败检测

### 不足

- 代码改造量比状态条件更新更大
- 对当前项目来说不一定必要

**建议**：
- 订单表：`where id=? and status=?` 就够用；
- 支付流水表：可加 `version`，用于演示你对乐观锁的理解。

## 7.3 必补唯一索引

### 订单请求表

- `uk_request_no`

### 支付流水表

- `uk_out_trade_no`
- `uk_channel_txn_no`

### 支付回调表

- `uk_callback_id`

没有唯一索引，幂等就没有物理约束，只能算“逻辑约定”。

## 8. 状态机如何融入

现有项目已具备订单状态机，因此不要推翻，而要把它升级为“第二道门”。

## 8.1 状态机负责什么

- 定义订单状态流转规则
- 防止非法迁移
- 统一支付、取消、接单、配送、完成等事件入口

## 8.2 Redis 锁负责什么

- 防止同一订单在同一时刻被多个线程并发推进

## 8.3 数据库 CAS 负责什么

- 防止多个线程在极端情况下同时写成功

### 最终职责划分

- **第一道门：Redis 锁** —— 拦并发
- **第二道门：状态机** —— 验状态
- **第三道门：数据库 CAS** —— 保最终一致性

这句话很适合你在面试时直接说。

## 9. 推荐技术选型结论

## 9.1 最推荐的落地组合

### 下单幂等

- `submitToken`：Redis + Lua
- 分布式锁：Redisson `RLock`
- 请求号：雪花 ID / UUID
- 请求日志表：MySQL 唯一索引

### 支付幂等

- 支付请求锁：Redisson `RLock`
- 回调幂等：回调日志表 + 唯一索引
- 状态推进：Spring StateMachine
- 最终写入：MyBatis 条件更新

### 数据库保护

- 优先：`where id=? and status=?`
- 可选增强：`version` 乐观锁

### 异步扩展

- 订单超时关闭：RabbitMQ / 延迟队列（可选）
- 支付成功通知商家端：MQ / WebSocket（可选）

## 9.2 不推荐的方案

### 不推荐 1：只加 Redis 锁

问题：
- 锁释放后仍可能重复处理
- 无法识别请求重放
- 缺少数据库兜底

### 不推荐 2：直接引入 Seata

问题：
- `sky-take-out` 当前不是典型微服务跨库事务场景
- 成本高，收益小
- 面试官会追问全局事务边界、AT/TCC 原理，得不偿失

### 不推荐 3：只靠前端按钮置灰

问题：
- 没有后端保障
- 无法应对重试、重放、恶意请求

## 10. 建议的表结构新增

## 10.1 `order_request_log`

```sql
create table order_request_log (
  id bigint primary key auto_increment,
  request_no varchar(64) not null,
  user_id bigint not null,
  cart_digest varchar(128) not null,
  order_id bigint null,
  status varchar(16) not null,
  create_time datetime not null,
  update_time datetime not null,
  unique key uk_request_no (request_no)
);
```

## 10.2 `payment_txn`

```sql
create table payment_txn (
  id bigint primary key auto_increment,
  order_id bigint not null,
  order_no varchar(64) not null,
  request_no varchar(64) not null,
  out_trade_no varchar(64) not null,
  channel_txn_no varchar(64) null,
  pay_channel varchar(16) not null,
  pay_status varchar(16) not null,
  amount decimal(10,2) not null,
  version int not null default 0,
  create_time datetime not null,
  update_time datetime not null,
  unique key uk_out_trade_no (out_trade_no),
  unique key uk_channel_txn_no (channel_txn_no)
);
```

## 10.3 `payment_callback_log`

```sql
create table payment_callback_log (
  id bigint primary key auto_increment,
  order_no varchar(64) not null,
  out_trade_no varchar(64) not null,
  channel_txn_no varchar(64) null,
  callback_id varchar(128) not null,
  raw_payload text,
  process_status varchar(16) not null,
  create_time datetime not null,
  unique key uk_callback_id (callback_id)
);
```

## 11. 推荐实施顺序

## Phase 1：先补数据库兜底

1. 改订单更新 SQL：全部改成条件更新
2. 补支付流水唯一索引
3. 补支付回调日志表

**原因**：这是收益最大、改造最稳的一步。

## Phase 2：再补支付幂等

1. 引入 `payment_txn`
2. 支付接口加 Redis 锁
3. 回调加日志表幂等
4. 回调走状态机 + 条件更新

**原因**：支付链路风险最高，最值得先做。

## Phase 3：最后补下单幂等

1. 确认页生成 `submitToken`
2. 提交订单 Lua 校验并删除 token
3. 提交订单加 Redis 锁
4. 新增 `order_request_log`

**原因**：下单幂等重要，但支付幂等更容易被面试官追问，也更体现工程复杂度。

## Phase 4：可选异步增强

1. 订单超时未支付自动取消
2. 支付成功后异步通知商家端
3. 审计日志与监控埋点

## 12. 简历表达建议

### 12.1 精简版

基于 Redis 分布式锁、订单状态机与 MyBatis 条件更新，完成下单与支付链路的幂等改造：通过 submitToken + 业务锁防止重复下单，通过支付流水表与回调日志表实现支付回调幂等，并以状态机约束合法流转、数据库 CAS 兜底最终一致性。

### 12.2 强化版

针对外卖场景中用户重复点击下单、弱网重试及支付回调重复通知问题，设计并落地“Redis 锁控并发、StateMachine 控状态迁移、MySQL 条件更新保一致”的订单幂等方案；新增订单请求日志、支付流水、回调日志三类表结构，避免重复下单、重复支付和非法状态推进。

### 12.3 面试回答主线

可以按下面顺序讲：

1. **场景**：重复点击下单、重复支付回调
2. **问题**：会造成重复订单、重复扣款、订单状态错乱
3. **方案**：
   - 下单：submitToken + Redis 锁 + requestNo
   - 支付：支付流水 + 回调幂等 + 状态机
   - DB：条件更新 / 乐观锁
4. **结果**：保证同一订单只被成功支付一次，重复请求返回已处理结果，不再造成状态错乱

## 13. 最终推荐结论

如果只选一套最适合 `sky-take-out`、最适合简历包装、同时也最容易真实落地的方案，那么结论是：

### 方案主干

- **下单幂等**：`submitToken + Redis Lua + Redisson 锁 + order_request_log`
- **支付幂等**：`payment_txn + payment_callback_log + Redisson 锁 + StateMachine`
- **最终保护**：`update ... where id=? and status=?`

### 工程取舍

- **要**：Redis、状态机、唯一索引、条件更新
- **可选**：RabbitMQ 延迟关单
- **不要硬上**：Seata

### 一句话总结

这不是“加了个 Redis 锁”，而是一套 **“请求级幂等 + 状态级约束 + 存储级兜底”** 的订单并发一致性方案。
